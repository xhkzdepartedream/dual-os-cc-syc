# Dual-OS Context Sync —— 使用指南

## 1. 适用场景

你在同一台机器上安装了 Windows + Linux 双系统。在 Windows 上使用 Claude Code（VS Code 插件）写代码、对话,切换回 Linux 后想接着之前的对话继续,但两个系统的 Claude Code 数据在不同分区、路径格式也不一样。

本工具负责:

- **文件同步**: 将最新一端的对话记录完整复制到对端
- **路径转换**: 自动把 `.jsonl` / `.json` 文件里的 Windows 绝对路径（如 `E:\DL\vggt`）替换为 Linux 绝对路径（如 `/home/xhkzdepartedream/.data2/DL/vggt`），反之亦然
- **安全保障**: 只读检测、脑裂保护、原子写入

---

## 2. 工作原理

```
关机前触发 (Linux systemd ExecStop)
        │
        ▼
 ┌──────────────────┐
 │ ① 扫描两端目录    │  ← 比较 .jsonl / .json 文件的 mtime
 │ ② 判断同步方向    │  ← auto_newest: 哪端新从哪端推
 │ ③ 安全检查        │  ← 可写性 / 脑裂检测
 │ ④ 复制到 .tmp_sync│  ← 临时目录,避免覆盖中断
 │ ⑤ 内容替换        │  ← 解析 JSON → 替换路径 → 重新序列化
 │ ⑥ 原子替换        │  ← os.replace() 原子 swap
 │ ⑦ 更新状态文件    │  ← 记录本次同步的时间戳和 mtime
 └──────────────────┘
```

**建议的时序**:

| 时机 | OS | 方向 |
|------|-----|------|
| Linux 关机前 | Linux | Linux → Windows（把本端新对话推给 Windows） |
| Windows 关机前 | Windows | Windows → Linux（把本端新对话推给 Linux） |

> 开机时同步也可以，但关机时"推送"比开机时"拉取"更自然——因为你知道关机时数据是最终状态。

## 3. 环境要求

- Python 3.10+
- Linux 端已挂载 Windows 分区至 `/mnt/c`, `/mnt/e` 等（`/etc/fstab` 配置）
- 安装依赖: `pip install pydantic pyyaml`

---

## 4. 配置文件 `config.yaml`

```yaml
settings:
  sync_strategy: "auto_newest"         # 自动判断方向
  state_file_path: "~/.config/dual_os_sync/sync_state.json"

access:
  linux_mount_win_c: "/mnt/c"          # Windows C 盘挂载点
  linux_mount_win_e: "/mnt/e"          # Windows E 盘挂载点
  win_mount_linux_root: null           # 一般不用

base_mappings:
  - id: dl_workspace
    win: "E:\\DL\\"                    # Windows 上的路径前缀
    linux: "/home/xhkzdepartedream/.data2/DL/"  # Linux 上的对应路径

tasks:
  - name: "Claude Code - VGGT"
    depends_on: "dl_workspace"         # 使用上面的映射规则
    win_path: "C:\\Users\\30413\\.claude\\projects\\e--DL--vggt"
    linux_path: "/home/xhkzdepartedream/.claude/projects/mnt-e--DL--vggt"
    target_extensions: [".jsonl", ".json"]
```

需要修改的地方:
- `xhkzdepartedream` → 你的 Linux 用户名
- `30413` → 你的 Windows 用户名
- 路径前缀按实际挂载点调整

## 5. 命令行用法

### 基本命令

```bash
# 执行全部同步任务（按 auto_newest 判定方向）
python -m dual_os_sync.cli -c config.yaml

# 只执行指定任务
python -m dual_os_sync.cli -c config.yaml -t "Claude Code - VGGT"

# 预览模式——只报告会做什么,不实际修改
python -m dual_os_sync.cli -c config.yaml --dry-run

# 强制指定方向（覆盖 auto_newest）
python -m dual_os_sync.cli -c config.yaml --force linux_to_win
python -m dual_os_sync.cli -c config.yaml --force win_to_linux

# 查看详细日志
python -m dual_os_sync.cli -c config.yaml -v
```

### 输出格式

```
[SYNCED] Claude Code - VGGT — /home/.../mnt-e--DL--vggt → /mnt/c/Users/.../e--DL--vggt
[OK]     Claude Code - VGGT — already in sync
[DRY-RUN] Claude Code - VGGT — would sync ... → ...
[CONFLICT] Claude Code - VGGT — split-brain detected, target backed up to .bak
[ERROR]   Claude Code - VGGT — target parent ... is read-only, aborting
[SKIP]    Claude Code - VGGT — neither side has data
```

## 6. 部署为自动服务

### 6.1 Linux（systemd 关机触发）

创建 `/etc/systemd/system/dual-os-sync.service`:

```ini
[Unit]
Description=Dual-OS Context Sync (shutdown trigger)
DefaultDependencies=no
Before=shutdown.target reboot.target halt.target

[Service]
Type=oneshot
RemainAfterExit=true
ExecStop=/usr/bin/python -m dual_os_sync.cli -c /home/xhkzdepartedream/.config/dual_os_sync/config.yaml
User=xhkzdepartedream
WorkingDirectory=/home/xhkzdepartedream/tools/sync

[Install]
WantedBy=multi-user.target
```

> `ExecStop` 在关机/重启时执行——这是关键。不要用 `ExecStart`。

激活:

```bash
sudo systemctl daemon-reload
sudo systemctl enable dual-os-sync.service
sudo systemctl start dual-os-sync.service   # 注册服务,但不执行同步
```

验证:

```bash
systemctl status dual-os-sync.service
journalctl -u dual-os-sync.service
```

### 6.2 Windows（任务计划程序 关机触发）

```powershell
# 以管理员身份运行 PowerShell
$Action = New-ScheduledTaskAction -Execute "python" `
    -Argument "-m dual_os_sync.cli -c C:\Users\30413\tools\sync\config.yaml --force win_to_linux" `
    -WorkingDirectory "C:\Users\30413\tools\sync"

$Trigger = New-ScheduledTaskTrigger -AtLogOn   # 或绑定事件 ID 1074 (关机)

Register-ScheduledTask -TaskName "DualOS Sync" `
    -Action $Action -Trigger $Trigger -RunLevel Highest
```

> Windows 端一般用 `--force win_to_linux`，因为 Windows 难以直接访问 ext4 分区。同步方向固定为"把 Windows 的新数据推到 Linux 挂载点"。

### 6.3 手动 alias（可选）

不想用自动服务的话,可以加到 `.bashrc`:

```bash
alias sync-claude='python -m dual_os_sync.cli -c ~/.config/dual_os_sync/config.yaml'
```

关机前手动执行一次即可。

## 7. 安全机制说明

| 机制 | 触发条件 | 行为 |
|------|----------|------|
| **只读检测** | 目标目录不可写（如 NTFS 脏挂载） | 立即中止,输出 `[ERROR]`,不写任何数据 |
| **脑裂保护** | 自上次同步后两端都被修改过 | 目标目录备份为 `.bak`,中止自动覆盖,输出 `[CONFLICT]` |
| **原子写入** | 每次同步 | 先写到 `.tmp_sync/` 临时目录,校验通过后 `os.replace()` 原子替换 |
| **二进制保护** | 非 `.jsonl` / `.json` 文件 | 仅文件拷贝,绝不执行正则替换 |

### 脑裂的手动处理

当出现 `[CONFLICT]` 时:

1. 目标目录已备份为 `<目录名>.bak`
2. 手动查看两端差异,决定保留哪一端
3. 删除 `.bak` 备份,或将其内容合并到目标目录
4. 再次运行同步工具,此时只有一端有修改,正常同步

## 8. 添加更多同步任务

在 `config.yaml` 的 `tasks` 下新增:

```yaml
tasks:
  - name: "Claude Code - VGGT"
    # ... 已有的配置 ...

  - name: "Claude Code - 另一个项目"
    type: "two_way_sync"
    depends_on: "dl_workspace"
    win_path: "C:\\Users\\30413\\.claude\\projects\\e--DL--another-project"
    linux_path: "/home/xhkzdepartedream/.claude/projects/mnt-e--DL--another-project"
    regex_replace: true
    target_extensions: [".jsonl", ".json"]
```

如果新项目的工作目录在另一个盘符（如 `D:\`），需要在 `base_mappings` 和 `access` 中新增对应的映射和挂载点。

## 9. 常见问题

### Q: 为什么 Windows 端不配置自动同步？

A: Windows 无法直接读写 Linux 的 ext4/btrfs 分区。如果需要 Windows→Linux 方向的同步，可以在 Windows 关机脚本中执行，前提是已通过 WSL 或 ext4 驱动挂载了 Linux 分区。

### Q: 同步后 Claude Code 能否立即继续对话？

A: 可以。`.jsonl` 和 `.json` 文件中的路径已被正确替换为当前系统的格式，Claude Code 重启后能直接识别项目路径。

### Q: 如果两端都修改了同一个会话怎么办？

A: 触发脑裂保护——旧端数据备份为 `.bak`，同步中止。你需要手动合并。建议不要在两端的同一个会话中同时工作。
