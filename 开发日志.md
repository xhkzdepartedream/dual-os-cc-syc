# 开发日志

## 2026-04-28

### 修复：路径配置错误导致 `[SKIP] neither side has data`

**现象**：两端都有数据，但同步工具输出 `[SKIP] Claude Code - VGGT — neither side has data`。

**排查过程**：

1. 配置中 `linux_mount_win_c` 为 `/home/xhkzdepartedream/windows_C`（大写 C），但实际 NTFS 挂载点是小写 `windows_c`。Linux 文件系统区分大小写，导致 Windows 端路径始终找不到。
2. `win_path` 写的是 `e--DL--vggt`（双连字符），但 Windows 上 Claude Code 实际生成的目录名是 `e--DL-vggt`（单连字符）。编码规则：盘符 `:` 变 `-`，每个 `\` 变 `-`，所以 `E:\DL\vggt` → `e--DL-vggt`。
3. 修正后仍然报错，调试发现 `resolve_win_path_on_linux` 的路径拼接有隐藏 bug：`win_path[2:]` 得到 `\Users\...`，`replace("\\", "/")` 后变成 `/Users/...`（以 `/` 开头）。Python `Path` 的 `/` 运算符在右侧为绝对路径时会丢弃左侧 mount 前缀。加上 `.lstrip("/")` 修复。

**修改的文件**：

- `config.yaml`：`windows_C` → `windows_c`；`e--DL--vggt` → `e--DL-vggt`；`linux_path` 改为正确的 Linux 项目目录名（`-home-xhkzdepartedream-data2-DL-vggt`）
- `dual_os_sync/config.py`：`resolve_win_path_on_linux` 中 `rest` 加 `.lstrip("/")`

### 修复：同步后状态记录旧 mtime 导致误判脑裂

**现象**：首次同步成功后，第二次运行检测到 split-brain，备份目标端后中止。

**原因**：`state.update()` 写入的是 `target_status.max_mtime`（同步前的目标端 mtime）。但 `atomic_copy` 用新文件替换了目标端，mtime 变为当前时间，与 state 记录值不一致。下次运行时 `determine_direction` 看到目标端 mtime 比记录值新，误判为目标端有变更。如果两端都被判定有变更，触发脑裂保护。

**修改**：`sync_engine.py` 中 `_sync_one()`，在 `atomic_copy` 后重新扫描目标端，用实际 mtime 更新 state。

**修改的文件**：

- `dual_os_sync/sync_engine.py`：增加 `scan_side(target_dir)` 重扫后写入 state

### 新增：对话内容中嵌入路径的正则替换

**现象**：同步后的 JSONL 文件中，结构化字段（如 `project_dir`）的路径已被正确替换，但 `<ide_selection>`、`<ide_opened_file>` 标签内以及用户消息正文中的路径仍然是 Windows 格式。

**原因**：原 `_replace_str` 只做前缀匹配（`starts_with`），只有字符串值本身就是路径时才生效。当路径出现在文本中间（如 "从 `E:\DL\vggt\main.py` 中"）时不被处理。

**修改**：`content_replacer.py` 新增 `_replace_embedded_paths` 函数，对每个 base mapping 的正反斜杠两种变体做正则搜索（`re.sub`），匹配路径前缀后跟随的路径字符序列（字母、数字、`_`、`-`、`.`、`/`、`\`），自动停在中文字符、空格、引号、尖括号等非路径字符前。同时支持 Linux → Windows 的反向替换。在 `_replace_str` 中前缀匹配失败后 fallthrough 调用。

**修改的文件**：

- `dual_os_sync/content_replacer.py`：新增 `import re`、`_replace_embedded_paths` 函数；`_replace_str` 末行改为 fallthrough 调用

### 新增：自动发现同步任务（`--auto` 模式）

**背景**：之前每个同步任务需要手动在 `config.yaml` 中写 `tasks`，指定 `win_path` 和 `linux_path`。Claude Code 的项目目录名已经编码了工作目录路径，JSONL 文件中也有 `cwd` 字段，足够自动化。

**实现**：

1. `auto_discover.py`（新文件，~280 行）：
   - `read_cwd()`：遍历项目目录所有 JSONL 文件的所有行，搜索 `cwd` 字段
   - `resolve_linux_path()`：将 Windows cwd 通过 base_mappings 或 mount 配置转为 Linux 路径
   - `infer_base_mapping_from_pair()`：从配对的 cwd 中自动推断路径映射关系（匹配最长路径后缀，提取前缀）
   - `auto_discover()`：扫描两端 `.claude/projects/` → 读取 cwd → 配对 → 推断 mapping → 生成 TaskConfig

2. `cli.py`：新增 `--auto` 标志。启用后调用 `auto_discover()`，注入推断的 base_mappings，替换 tasks

3. `config.yaml`：新增 `linux_mount_win_d` 和 `linux_mount_win_e` 字段

**多处修复**：

- `resolve_win_path_on_linux` 需 `.lstrip("/")`，否则 Path 拼接会丢弃 mount 前缀
- `mount_map` 改为动态从所有 `linux_mount_win_X` 字段构建
- `TaskState` 改用 `last_win_mtime`/`last_linux_mtime`（而非方向相关的 source/target），兼容旧格式
- 自动发现的 `win_path` 必须用 Windows 原生格式（`C:\Users\...`），不能用 Linux 解析路径

**修改的文件**：
- `dual_os_sync/auto_discover.py`（新建）
- `dual_os_sync/cli.py`：增加 `--auto` 标志处理
- `dual_os_sync/config.py`：增加 `linux_mount_win_d`、`linux_mount_win_e`、`_mount_map()`、`resolve_linux_path_on_windows`
- `dual_os_sync/state.py`：`TaskState` 改为 per-side mtime 字段，兼容旧格式
- `dual_os_sync/sync_engine.py`：state update 按同步方向条件赋值

### 修复：跨文件系统 mtime 精度差异导致反复同步

**现象**：D 盘的 `\tools\sync` 项目每次运行都重新同步，其他项目正常。

**原因**：`determine_direction` 使用严格 `>` 比较 mtime。ntfs3 内核驱动在 FILETIME ↔ timespec64 转换中引入亚微秒级精度误差（约 100ns 量级）。实际写入的文件 mtime 与 state 记录值有微小差异，导致 `>` 始终为 True。

**修改**：`direction.py` 中增加 `_MTIME_EPSILON = 1.0`，比较时用 `> last_mtime + EPSILON`。1 秒的容差足够覆盖 ntfs3 和 FUSE 挂载的精度差异，同时不会掩盖真正的文件变更。

**修改的文件**：
- `dual_os_sync/direction.py`：新增 `_MTIME_EPSILON` 常量；win_changed/linux_changed 比较加 epsilon

### 修复：未配置 base_mapping 的驱动器路径未被内容替换

**现象**：同步后的 JSONL 中仍有 `cd 'C:\Users\30413' && claude --resume xxx` 等 Windows 原生命令。

**原因**：内容替换只使用 `base_mappings` 中配置的路径映射（如 `E:\DL\`）。`C:\Users\30413` 等路径没有对应的 base_mapping，即使 access 配置了 C 盘挂载点也不参与替换。

**修改**：`config.py` 中新增 `AccessConfig.drive_mappings()`，根据 `linux_mount_win_*` 配置自动生成 `C:\` ↔ `/home/xhkzdepartedream/windows_c/` 等驱动器级别的 BaseMapping。`get_mappings_for_task()` 在返回任务特定映射后追加这些驱动器映射作为 fallback。

**修改的文件**：
- `dual_os_sync/config.py`：新增 `DRIVE_LETTERS`、`drive_mappings()` 方法；`get_mappings_for_task()` 末尾追加驱动器映射

### 新增：`--force-sync` 强制同步标志

**背景**：用户删除某端 `.claude/projects/` 中的数据后，由于 state 记录的 mtime 未变，`determine_direction` 判定 IN_SYNC 不执行同步。

**修改**：新增 `--force-sync` 命令行标志。启用后忽略保存的 state（将 `last_sync_ts`、`last_win_mtime`、`last_linux_mtime` 全部设为 None），让 `determine_direction` 重新根据两端实际数据决定同步方向。不改变 `force_*` 策略行为。

**用法**：
```bash
python -m dual_os_sync.cli -c config.yaml --auto --force-sync
```

**修改的文件**：
- `dual_os_sync/sync_engine.py`：`run_sync()` 和 `_sync_one()` 增加 `force_sync` 参数；`force_sync=True` 时忽略 state 调用 determine_direction
- `dual_os_sync/cli.py`：增加 `--force-sync` 参数并传递至 run_sync
