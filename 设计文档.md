## 1. 数据结构分析（基于实际样本）

```
    │   ├── agent-{uuid}.jsonl       # 子 agent 对话（107行）
    │   └── agent-{uuid}.meta.json   # 子 agent 元数据（小文件）
```

### 1.2 路径出现模式（原始 JSONL 文本中的实际字节）

以下示例来自真实文件 `39f4522e-5b28-4382-88da-327e0afefcae.jsonl`：

| JSON 字段 | 原始文本中的形式 | JSON 解码后的值 |
|-----------|-----------------|----------------|
| `cwd` | `"cwd":"e:\\DL\\vggt"` | `e:\DL\vggt` |
| `filePath` | `"filePath":"e:\\DL\\vggt\\docs\\VGGT-revised.md"` | `e:\DL\vggt\docs\VGGT-revised.md` |
| `filename` | `"filename":"e:\\DL\\vggt\\docs\\VGGT-revised.md"` | `e:\DL\vggt\docs\VGGT-revised.md` |

### 1.3 关键发现

- **盘符大小写**：绝大多数是 **小写 `e:`**，极少数 `E:`（每个文件中 ~2 次），必须**大小写不敏感匹配**
- **JSON 转义**：原始文本中单个 `\` 被 JSON 编码为 `\\`（两个字符），正则替换必须匹配 `\\` 形式的文本
- **分隔符**：Windows 路径在原始文本中使用 `\\` (JSON-escaped `\`)，替换为 Linux 路径后使用 `/`（无需转义）
- **无关误匹配**：消息正文（content 字段）包含大量 `\n`（JSON 中的换行符），会匹配 `字母:\字母` 模式但并非路径。策略：**仅在已知字段键的 JSON 字符串值内替换**，避免对 markdown 正文误伤

### 1.4 需要同步的文件类型

| 文件 | 类别 | 处理方式 |
|------|------|----------|
| `*.jsonl` | 文本 | 拷贝 + 路径内容替换 |
| `*.json` | 文本 | 拷贝 + 路径内容替换 |
| 其他（无扩展名等） | 二进制 | 仅拷贝，禁止内容替换 |

## 4. 内容替换详细设计

### 4.1 核心挑战

原始 JSONL 文本中的 Windows 路径是 JSON-escaped 形式：

```
原文: "cwd":"e:\\DL\\vggt"
                     ↑↑ JSON 转义的双反斜杠
```

一个朴素的 `E:\\DL\\` → `/home/user/.data2/DL/` 替换是正确的，因为：

- `E:\\DL\\` 恰好匹配 JSON 中的 `E:\DL\`
- 替换为 `/home/user/.data2/DL/`，正斜杠在 JSON 中无需转义

**但余下路径的 `\\` 需转为 `/`**。例如：
```
原始: e:\\DL\\vggt\\docs\\VGGT-revised.md
       ^^^^^^^^  ^^^^^^^^^^^^^^^^^^^^^^^^^
       映射替换   需转为 / 的部分
```

### 4.2 替换策略（两步法）

**Step 1 —— 识别并提取完整路径**：
- 在原始文本中匹配已知的路径模式：`(E:|e:)\\DL\\[^\"]+`（在 JSON 字符串值边界 `"` 前终止）
- 提取完整路径的 JSON-escaped 形式

**Step 2 —— 逐路径替换**：
- 对每个唯一路径，计算其 Linux 等价形式
- 在原始文本中做精确字符串替换

算法伪代码：
```python
def build_replacement_map(raw_text: str, base_mappings: list) -> dict:
    """从原始文本中提取所有 Windows 路径，生成替换映射表。"""
    replacements = {}
    for pattern in find_windows_paths_in_json(raw_text):
        linux_path = convert_path(pattern, base_mappings)
        replacements[pattern] = linux_path
    return replacements

def convert_path(json_escaped_win_path: str, mappings) -> str:
    """将 JSON-escaped 的 Windows 路径转为 Linux 路径。"""
    # 1. JSON 解码: e:\\DL\\vggt\\docs  → e:\DL\vggt\docs
    decoded = json_escaped_win_path.replace("\\\\", "\\")  # noqa
    # 2. 大小写不敏感匹配基础映射并替换前缀
    for m in mappings:
        decoded = re.sub(m.win_pattern, m.linux, decoded, flags=re.IGNORECASE)
    # 3. 剩余反斜杠转正斜杠
    decoded = decoded.replace("\\", "/")
    return decoded
```

### 4.3 非路径内容的保护

路径替换**仅在 JSON 结构性字段的值中进行**（`cwd`, `filePath`, `filename`），不对 markdown 正文 (`content` 字段) 做替换。理由：

- 正文中的路径引用（如 `e:\DL\vggt`）作为自然语言文本存在，不影响 Claude Code 在 Linux 上的功能
- 误替换正文内容可能导致数据损坏
- 精确字段匹配避免误伤 `\n` 等 JSON 转义字符

## 5. 安全机制

### 5.1 只读检测
同步前对目标路径执行 `os.access(path, os.W_OK)` 检测，不可写时立即中止并输出错误日志。处理 Windows NTFS 脏挂载（未彻底关机导致 Linux 侧只读挂载）场景。

### 5.2 脑裂保护
- 读取两端的 `sync_state.json`
- 若自上次同步后，两端各自的最新 mtime 都大于记录的同步时间 → 判定为脑裂
- 脑裂行为：将目标端旧文件备份为 `.bak`，中止自动覆盖，输出冲突日志

### 5.3 原子写入
- 先将新文件写入目标目录旁的 `.tmp_sync/` 临时目录
- 校验临时目录完整性（文件数匹配、无损坏）
- `os.replace()` 原子替换（同文件系统内为 rename，保证原子性）
